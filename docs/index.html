1. ¿Qué es y para qué sirve?1. What is it and what is it for?

El conector GitLab permite que el agente de Ruvic gestione issues, comente en Merge Requests, y revise el estado de tus pipelines de CI/CD, directamente en tus proyectos de GitLab. Una vez configurado, puedes pedirle al agente cosas como:

  • “Crea un issue reportando este bug que me describieron.”
  • “¿Qué issues abiertos tenemos en el proyecto X?”
  • “Comenta en el MR 42 que se ve bien.”
  • “¿Pasó el pipeline de CI en la rama main?”

Operaciones soportadas:

The GitLab connector lets the Ruvic agent manage issues, comment on Merge Requests, and check your CI/CD pipeline status, directly on your GitLab projects. Once configured, you can ask the agent things like:

  • “Create an issue reporting this bug I was told about.”
  • “What open issues do we have in project X?”
  • “Comment on MR 42 that it looks good.”
  • “Did the CI pipeline pass on the main branch?”

Supported operations:

Fuera de alcance a propósitoOut of scope by design Este conector no puede fusionar Merge Requests, eliminar proyectos o ramas, ni hacer force-push. These are potentially destructive actions left out by design.

2. Prerrequisitos2. Prerequisites

  • Una cuenta de GitLab (gitlab.com, o una instancia self-managed) con acceso de al menos Developer al/los proyecto(s) que se van a usar.
  • Para pruebas: usa un proyecto de pruebas propio en vez de un proyecto productivo real.
  • A GitLab account (gitlab.com, or a self-managed instance) with at least Developer access to the project(s) to be used.
  • For testing: use your own test project instead of a real production project.

3. Configuración en el fabricante (paso a paso)3. Vendor-side setup (step by step)

  1. Ve a la configuración de Access Tokens.

    En GitLab, clic en tu foto de perfil (arriba a la derecha) → Edit profile → en el menú lateral, Access Tokens.

    Go to Access Tokens settings.

    In GitLab, click your profile picture (top right) → Edit profile → in the sidebar, Access Tokens.

  2. Crea un token nuevo.

    Ponle un nombre descriptivo, ej. "Ruvic GitLab Connector", y una fecha de expiración (recomendada).

    Create a new token.

    Give it a descriptive name, e.g. "Ruvic GitLab Connector", and an expiration date (recommended).

    [ Captura: formulario "Add new token" — reemplazar por assets/img/gl-new-token.png ] [ Screenshot: "Add new token" form — replace with assets/img/gl-new-token.png ]
  3. Selecciona los scopes.

    Marca api (acceso completo) para simplicidad, o si prefieres acotar más: read_api + read_repository como mínimo para las operaciones de lectura, agregando write_repository si vas a crear issues/comentar.

    Select the scopes.

    Check api (full access) for simplicity, or if you prefer to scope it down more: read_api + read_repository at minimum for read operations, adding write_repository if creating issues/commenting.

  4. Crea y copia el token.

    Clic en "Create personal access token". Cópialo de inmediato — GitLab solo lo muestra completo una vez.

    Create and copy the token.

    Click "Create personal access token". Copy it immediately — GitLab only shows it in full once.

    Riesgo de seguridadSecurity risk Cualquiera que tenga este token puede actuar en tu nombre dentro del alcance otorgado. Trátalo como una contraseña. Anyone with this token can act on your behalf within the granted scope. Treat it like a password.
  5. Confirma que el usuario tenga rol Developer (o superior) en el proyecto.

    El scope del token no es suficiente por sí solo — el usuario dueño del token también necesita rol de al menos Developer en cada proyecto donde el agente vaya a crear issues o comentar. Revísalo en Project → Manage → Members.

    Confirm the user has Developer role (or higher) on the project.

    The token's scope alone isn't enough — the token's owner also needs at least Developer role on each project where the agent will create issues or comment. Check this in Project → Manage → Members.

  6. Prueba desde línea de comandos (opcional pero recomendado).
    Test from the command line (optional but recommended). 
    curl --header "PRIVATE-TOKEN: TU_TOKEN" \
  "https://gitlab.com/api/v4/user"

    Una respuesta con tus datos de usuario confirma que el token funciona.

    A response with your user data confirms the token works.

5. Configuración en Ruvic5. Setup in Ruvic

Entra a Settings → Conectores, busca la tarjeta GitLab y pulsa Configurar. Pega el token en el campo Access Token, deja los demás campos por defecto salvo que uses una instancia self-managed.

Go to Settings → Connectors, find the GitLab card and click Configure. Paste the token into the Access Token field, leave the rest as default unless you use a self-managed instance.

Guarda y pulsa Probar conexión.

Save and click Test connection.

6. Verificación6. Verification

  1. Test de conexión exitoso.

    El botón Probar conexión debe responder: "Conexión exitosa a GitLab (https://gitlab.com/api/v4)".

    Successful connection test.

    The Test connection button should return: "Successful connection to GitLab (https://gitlab.com/api/v4)".

  2. Prueba mínima con el agente.

    Pide algo de solo lectura primero:

    "¿Qué issues abiertos hay en el proyecto X?"
    Minimal test with the agent.

    Ask something read-only first:

    "What open issues does project X have?"

8. Preguntas frecuentes y notas de seguridad8. FAQ and security notes

¿El agente puede fusionar Merge Requests o eliminar proyectos?Can the agent merge Merge Requests or delete projects?

No. Esta versión del conector no incluye esas acciones, a propósito — son potencialmente destructivas o requieren juicio humano.

No. This version of the connector doesn't include those actions, by design — they're potentially destructive or require human judgment.

¿Dónde queda guardado el token?Where is the token stored?

Como secreto en el almacén de credenciales de la plataforma. La interfaz nunca vuelve a mostrarlo.

As a secret in the platform's credential store. The UI never displays it again.

Rotación de credencialesCredential rotation

Si le pusiste fecha de expiración al token (recomendado), genera uno nuevo antes de que venza (sección 3) y actualízalo en Ruvic. GitLab también permite "rotar" un token desde su propia página de detalle, generando uno nuevo con el mismo alcance.

If you set an expiration date on the token (recommended), generate a new one before it expires (section 3) and update it in Ruvic. GitLab also allows "rotating" a token from its own detail page, generating a new one with the same scope.

¿Qué hago si el token se compromete?What if the token is compromised?

  1. Ve a Access Tokens en GitLab y revoca el token de inmediato.
  2. Genera uno nuevo con el mismo alcance.
  3. Actualízalo en el formulario del conector.
  1. Go to Access Tokens in GitLab and revoke the token immediately.
  2. Generate a new one with the same scope.
  3. Update it in the connector form.
Buenas prácticasBest practices Usa el scope más acotado posible; pon fecha de expiración; nombra el token de forma descriptiva; y verifica que el rol del usuario en cada proyecto sea el mínimo necesario (Developer, no Maintainer/Owner, salvo que se requiera). Use the narrowest scope possible; set an expiration date; name the token descriptively; and verify the user's role on each project is the minimum needed (Developer, not Maintainer/Owner, unless required).
